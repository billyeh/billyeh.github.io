Host Your Own PocketBase (or Any Local Server) Online for Free with DuckDNS and Caddy

Want to make your local server (like PocketBase) accessible on the internet—securely, with HTTPS—without paying for a domain or hosting? This guide shows you how to do exactly that using DuckDNS and Caddy on macOS (though it works similarly on Linux and Windows).

🧩 Overview

We’ll set up:

  • DuckDNS – Free dynamic DNS service (maps a free subdomain to your public IP)
  • Caddy – Lightweight web server and reverse proxy with automatic HTTPS
  • PocketBase – Example app running locally on port 8090

Once done, your site will be available at:

https://<your-domain>.duckdns.org/

1️⃣ Create a DuckDNS Subdomain

  1. Go to https://www.duckdns.org
  2. Sign in with GitHub, Google, or another provider.
  3. Under domains, create a name — e.g. myproject.
  4. Copy your token (you’ll need it for DNS-based SSL).
  5. Note your chosen domain:
myproject.duckdns.org

2️⃣ Point DuckDNS to Your Current IP

Normally, DuckDNS automatically sets your domain to the public IP of the machine making the update request — no need to specify it manually.

Run this once to register your current IP:

curl "https://www.duckdns.org/update?domains=myproject&token=<your-token>"

You should see:

OK

To confirm it worked:

nslookup myproject.duckdns.org 8.8.8.8

You should see your public IP (check it with whatismyipaddress.com ).

3️⃣ Install PocketBase (example app)

Download and unpack PocketBase (or your own binary):

curl -L https://github.com/pocketbase/pocketbase/releases/latest/download/pocketbase_darwin_arm64.zip -o pb.zip
unzip pb.zip
./pocketbase serve --http=0.0.0.0:8090

💡 Using 0.0.0.0 instead of the default localhost (127.0.0.1) allows external users to reach the server.

PocketBase runs on http://127.0.0.1:8090 by default.

4️⃣ Install Caddy with DuckDNS Support

a. Download Caddy with DuckDNS DNS plugin

Visit the Caddy Download Page and under “Build Custom”, select:

  • Plugin: tls.dns.duckdns
  • Platform: macOS arm64 (for M1/M2 Macs)

Then unzip and move the binary:

sudo mv caddy_darwin_arm64_custom /usr/local/bin/caddy
sudo chmod +x /usr/local/bin/caddy

Confirm it runs:

caddy version

5️⃣ Configure Caddyfile

Create Caddyfile in the same directory:

myproject.duckdns.org {
    reverse_proxy localhost:8090

    tls {
        dns duckdns <your-duckdns-token>
    }
}

This tells Caddy to:

  • Listen on HTTPS for myproject.duckdns.org
  • Proxy requests to your local app on port 8090
  • Obtain and renew certificates automatically via DuckDNS

6️⃣ Run Caddy

Run:

sudo caddy run --config ./Caddyfile --adapter caddyfile

You should see output similar to:

INFO    serving initial configuration
INFO    http.enabling automatic TLS certificate management
INFO    http.log server running {"name": "srv0", "protocols": ["h1", "h2"]}

✅ This means Caddy is running and managing HTTPS certificates.

7️⃣ Configure Router Port Forwarding

You need to expose ports 443 (HTTPS) and optionally 80 (HTTP) to your Mac’s internal IP.

For Google WiFi / Nest routers

  1. Open the Google Home app
  2. Tap Wi-Fi → Settings → Advanced Networking → Port Management
  3. Add a new rule:
  • Internal IP: your Mac’s local IP (e.g. 192.168.86.42)
  • External port: 443
  • Internal port: 443
  • Protocol: TCP
  • Device: select your Mac

If port 80 gives you “failed to add new rule”, skip it — HTTPS (443) is enough.

8️⃣ Verify It’s Live 🌍

Open your browser to:

https://myproject.duckdns.org

If everything’s configured, you’ll see your PocketBase UI.

✅ Success Checklist

Step Check
DuckDNS domain resolves to your IP ✅ nslookup matches your public IP
PocketBase runs locally ✅ open http://localhost:8090
Caddy runs without errors ✅ “serving initial configuration” message
HTTPS works externally ✅ Browser loads site with lock icon

🧰 Troubleshooting

Port forwarding issues
  • If port 80 or 443 can’t be added, try deleting other rules first.
  • Some ISPs block port 80 — that’s fine, Caddy can still serve HTTPS on 443.
  • Test your open ports with https://www.canyouseeme.org.
Domain resolves but won’t connect 
ping myproject.duckdns.org
  • Ensure your local firewall allows incoming on 443.
  • Confirm Caddy is still running.
  • Verify your Mac’s local IP matches the port forwarding rule.
Caddyfile parse error: “wrong argument count or unexpected line ending after 'h1'”

Make sure your Caddyfile doesn’t contain any trailing spaces or comments in the tls section. It should look exactly like:

myproject.duckdns.org {
    reverse_proxy localhost:8090
    tls {
        dns duckdns <your-token>
    }
}
ERR_QUIC_PROTOCOL_ERROR in Chrome

That’s an HTTP/3 issue, usually harmless. Try a hard refresh or disable QUIC in Chrome:

chrome://flags/#enable-quic → Disabled

🔄 Keeping It Running

If you restart your Mac or your IP changes:

  1. Update your IP on DuckDNS: 
    curl "https://www.duckdns.org/update?domains=myproject&token=<your-token>&ip="
  2. Restart Caddy: 
    sudo caddy run --config ./Caddyfile --adapter caddyfile
  3. Visit your domain again.

For persistent setup, consider using Launchd (macOS) or systemd (Linux) to auto-start Caddy on boot.

🧠 Recap

  • A free dynamic DNS with DuckDNS
  • A reverse proxy and HTTPS certificate manager with Caddy
  • A secure, internet-accessible PocketBase instance

This pattern applies to any local web service — Node, Flask, Django, Go, etc.

Example Final Architecture

[Internet] ⇄ [DuckDNS Domain] ⇄ [Router Port 443] ⇄ [Caddy] ⇄ [PocketBase on 8090]
